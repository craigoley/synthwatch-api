# synthwatch-api ‚Äî Claude rules

Rules Claude should follow when working in this repo.

## Auth gate decision rule (copy this when adding an endpoint)

**Mutating verb (POST/PUT/PATCH/DELETE)?** Do nothing ‚Äî the middleware verb-gate covers it automatically
(fail-closed: editor/admin session required; `/editors*`/`/access-requests*` are admin-floor in the gate;
`/auth/logout` is session-floor). **GET that serves credentials, forensic artifacts, or operator config?**
Add a handler self-guard with the `RequireSessionAsync` pattern (#154) ‚Äî the verb-gate never sees reads.
**User management or anything that must hold even mid-rollout?** Make the handler guard ignore the flag
(the Editors pattern). `AUTH_ENFORCEMENT_ENABLED` exists for **deploy safety only**: it is FAIL-CLOSED
(ON unless explicitly `false`/`0`), turning it off opens every write, all paid-AOAI endpoints, and
reconcile/apply at once, and startup logs a warning while it is off ‚Äî never design an endpoint's security
around the flag being off.

## Lessons from 2026-06-29

- **The runner (synthwatch repo) OWNS the DB schema + migrations; this API serves reads/writes against it.** API entity/DbContext changes must match columns the runner's migrations create, and any validation gate should MIRROR the runner's canonical rule (`reconcile.ts`, `tags.ts`, `locations.ts`) exactly ‚Äî as a shared pure helper with a cross-ref comment + predicate-parity `[InlineData]` tests so the two stay in sync. *(from #114 ‚Äî `CheckValidation.SensitiveNeedsRedaction` mirrors `reconcile.ts:181`; same pattern as the setCheckTags/setCheckLocations mirrors and the "cross-repo contract fix" commits)*

- **Merged ‚âÝ migrated.** This API auto-deploys on merge. Before merging API code that reads a NEW column, verify the runner's migration is actually APPLIED in the target env (`SELECT max(version) FROM schema_migrations`, or confirm the column exists) ‚Äî not merely merged in the runner repo ‚Äî or the deploy crashes querying a non-existent column. *(from #114 ‚Äî runner #137/migration 0046 was merged to runner main while prod was still at 0045 mid-task; the read-only mapping was only deploy-safe once 0046 was applied)*

- **CodeQL `paths-ignore` is a silent no-op for compiled C#.** The manual `dotnet build` compiles `obj/.../*.g.cs` (Functions Worker SDK + regex source generator), so the extractor flags them regardless of the globs; `query-filters` can't scope by path either. The working exclusion is a post-analysis `advanced-security/filter-sarif` step (filters by result location). *(from #113)*

- **CodeQL triage discipline:** dismiss only with a cited reason from {proven false-positive, won't-fix (tests/generated), mitigated-elsewhere}; never dismiss a security-category finding unless proven FP; split a rule group if only some instances are real (generated-file alerts carry `classifications:["generated"]`). The GitHub code-scanning `dismissed_comment` is **capped at 280 chars** (HTTP 422 otherwise) ‚Äî keep justifications terse. Note: `ExecuteSqlInterpolatedAsync($"‚Ä¶{x}‚Ä¶")` binds args as parameters (FormattableString), so CodeQL "ToString on FormattableString" / SQL-injection flags on those are false positives. *(from #111/#112/#113)*

- **ASP.NET Functions route params arrive ALREADY URL-decoded** (`ConfigureFunctionsWebApplication` decodes before binding) ‚Äî don't add `Uri.UnescapeDataString` (double-unescape corrupts URL-encoded emails). For set-based deletes use `ExecuteDeleteAsync`, not load-then-`RemoveRange`-then-`SaveChanges` (the tracker/audit path 500'd). *(from the #104‚Üí#105‚Üí#106‚Üí#107 access-requests delete chain ‚Äî #104 guessed "missing decode" and was the wrong fix; #106 found the real cause)*

- **ARM/ACA job-start POSTs (`Microsoft.App/jobs/start`) require `Content-Type: application/json` even with an empty body** ‚Äî a `text/plain` empty body returns 415 and the job never fires. *(from #101 ‚Äî the on-demand "Run now"/reconcile trigger silently no-op'd until this)*

- **The reconcile/run-now triggers fire AS the API's system-assigned managed identity** (`DefaultAzureCredential` in `Program.cs`, not admin/passed creds). To PROVE a MI-gated action, hit the DEPLOYED endpoint authed so the start runs under the function's MI ‚Äî running `az` as admin proves nothing about the MI. The MI needs `Container Apps Jobs Operator` (covers `Microsoft.App/jobs/start/action`) on the job; codify the grant in `infra/main.bicep`, don't leave it as a manual CLI assignment. *(from #108/#115)*

- **To authenticate as admin for a live proof, mint a `sessions` row directly:** `token_hash` = lowercase sha256-hex of an opaque token, `email` = an `ADMIN_EMAILS` entry, set `expires_at`, then send `Authorization: Bearer <token>`; delete the row afterward. *(from #115 reconcile-trigger proof)*

- **"Stale rows on a 200" from the runs/incidents list endpoints is usually NOT server caching.** These endpoints window by `started_at < to` where `to` defaults to request-time `now`; there is no response cache (scoped DbContext + `AsNoTracking`), so identical real round-trips returning a frozen newest-id is typically the CLIENT freezing the `to` param. Localize the layer with the psql-vs-curl test: run the endpoint's exact query in psql and curl the live endpoint at the same instant. *(from the runs-endpoint-stale-rows recon ‚Äî the "API is stale" premise was actually the dashboard's `useDateRange` freezing `to` at mount)*

- **Build/verify gate:** `dotnet build -warnaserror` (expect 0/0) + the Testcontainers xUnit suite (needs Docker running). Recon-first earns its keep here ‚Äî multiple tasks' stated root cause was wrong (B10 "already landed" wasn't; "the API returns stale rows" was a client bug), so verify a task's asserted layer from code + a live test BEFORE building the fix. *(from #114 and the B10 / reconcile / runs-stale recon tasks)*

## Lessons from 2026-07-02

- **`runs.retry_count` is an ATTEMPT count (runner migration 0048): `1` = first try / NO retry; an ACTUAL retry is `retry_count > 1`, and `0` never occurs in real data.** Any "was this run retried?" logic must filter `retry_count > 1`, never `> 0` (which counts every clean pass as retried). *(from #152 ‚Äî #149's `/trust` SQL used `> 0`, so every healthy monitor showed "flaky"; the fix flipped it to `> 1`. ‚òÖ Worse: the test seeded `retry_count=1` to mean "retried" ‚Äî the SAME inverted assumption as the code ‚Äî so the suite was 328/328 green while prod was wrong. Ground a fixture in what the value MEANS in prod, not in the code's reading of it.)*

- **`HttpRequest.ReadFromJsonAsync<T>()` throws `System.InvalidOperationException` (NOT the `JsonException` handlers catch) on a wrong/absent `Content-Type`** ‚Äî so a `catch (JsonException)` misses it and it becomes a shielded 500. Read JSON bodies through the shared `Infrastructure/RequestJson.ReadAsync<T>` helper: it guards `HasJsonContentType()` ‚Üí 415 first, then catches `JsonException` ‚Üí 400. Don't hand-roll a bare `ReadFromJsonAsync` + `catch (JsonException)`. *(from #155 ‚Äî every body endpoint 500'd on `Content-Type: text/plain`.)*

- **The AuthorizationMiddleware verb-gate returns `Allow` for EVERY GET (reads are open by default) and only sets `http.Items["principal"]` for audited MUTATIONS** ‚Äî so a GET that must be authenticated (the forensic-artifact endpoints) CANNOT lean on the middleware; it must self-guard by resolving `IAuthPrincipal.FromBearerAsync` and requiring `principal.CanWrite` (editor/admin), gated on `AuthorizationMiddleware.EnforcementEnabled()`. Require the role FLOOR, not just `principal != null` ‚Äî a revoked editor's still-valid session resolves to `anonymous` and would pass a null-check. *(from #154 ‚Äî the four artifact endpoints leaked forensic data anonymously; review caught the any-session-vs-editor/admin gap ‚Üí added the `CanWrite` floor.)*

- **The Testcontainers fixture `tests/SynthWatch.Api.Tests/fixtures/schema.sql` is a hand-maintained snapshot that LAGS live prod** (the inverse of "merged ‚âÝ migrated" ‚Äî here the TEST env trails). It has been missing columns/enum values that exist in prod: `runs.spec_provenance`, and `infra_error` in the `runs_status_check` CHECK. Before writing a test that needs a column or CHECK value, confirm the fixture has it and add it to mirror prod, or the seed SQL fails. *(from #153 ‚Äî added `infra_error` to seed an errored run; the trust work similarly had to add `spec_provenance`.)*

- **A mutating POST/PUT/DELETE 401s at the AuthorizationMiddleware BEFORE the handler runs when `AUTH_ENFORCEMENT_ENABLED` is on.** To reproduce a HANDLER-level bug against the deployed API, hit an unauth-allowlisted endpoint (`/auth/request-code|verify|request-access`) or send a valid bearer ‚Äî an anonymous curl to a gated mutating route proves nothing about the handler and returns a misleading 401. *(from #155 ‚Äî the task's suggested repro `POST /checks/parse-intent` returned 401 for every content type and never reached the body-parse bug; the unauth `POST /auth/request-code` reproduced the 500.)*

- **Host matching against `deploys.target_host` needs query-side normalization** ‚Äî lowercase + strip a leading `www.` on BOTH sides (`regexp_replace(lower(host),'^www\.','')`). The runner's `hostOf()` stores `www.`, so an apex-host check (e.g. the `wegmans.com` SSL check) can never string-match a `www.wegmans.com` deploy. Normalize READ-side only; the runner's stored convention + its `(target_host, fingerprint)` dedup key are runner-owned and out of scope. *(from #157 ‚Äî the incident deploy-proximity match.)*

## Lessons from 2026-07-12

- **The "Runner column parity (fixture vs runner schema)" CI check is a recurring landmine ‚Äî and it IS BLOCKING: it is in ci-gate's `REQUIRED` list (`.github/workflows/ci-gate.yml`), so a red parity holds the merge.** (Earlier this rule said "NOT blocking, main can sit red" ‚Äî that was stale; #167 promoted it into `REQUIRED` and it has gated ever since. The doc, the gate, and `schema-parity.yml`'s own header now agree: it blocks.) It replays the runner's migrations and whitespace-normalized-`pg_get_functiondef`-diffs the fixture's SHARED objects (columns AND function BODIES) against them. When it fails, the fix is a fixture bump copied **VERBATIM** from the runner's `db/schema.sql` (it's a text compare ‚Äî hand-retyping drifts). Confirm it's GREEN before assuming your PR is clean. **For a coordinated shared-schema change, the merge order is runner-first:** after the runner migration lands on runner-`main`, THIS fixture PR's parity flips green (it reads runner-`main` live) and then merges ‚Äî do not expect the fixture PR to be green before the runner PR merges. *(from #206 cost_projection alias drift, #207 sla_availability 0070 body, #224 the 0077 confirmation/superseded FK constraints; #226 hit a NEW pre-existing red ‚Äî 0069 cost_projection "cannot change return type" from #225's cost work ‚Äî unrelated to the PR, correctly reported not fixed; the NOT-blocking‚Üíblocking correction resolves a doc-vs-gate contradiction the #313‚ü∑#255 deadlock surfaced. Extends the 2026-07-02 "fixture LAGS prod" rule with the gate name + function-body compare + verbatim-copy discipline.)*

- **The trusted-author auto-merge fires on the branch state at merge-queue time and BYPASSES a red required check** ‚Äî so a fix commit pushed AFTER it triggers is ORPHANED (the squash captured the earlier commit) and main is left broken. VERIFY your fix is actually in the merge commit (`git show origin/main:<path> | grep <the fix>`); if main is red/incomplete, **fix-forward with a new PR** (don't assume the late commit joined). *(from #223 ‚Äî its fixture-FK fix commit was orphaned when auto-merge squashed the earlier commit ‚Üí main left red on parity ‚Üí fix-forwarded as #224; same shape as the earlier #200‚Üí#201.)*

- **`checks` has a `browser_needs_flow` CHECK: a `kind='browser'` row requires a non-null `flow_name`.** A test seed that inserts a browser check without `flow_name` fails with `23514`. Add `flow_name='<x>'`, or use `kind='http'` when the test is kind-agnostic. *(from #218, #223 test-seed fixes ‚Äî the same 23514 bit multiple test seeds this session.)*

- **C# embedded-string traps in test/SQL literals:** a literal `"` inside a `$@"‚Ä¶"` verbatim interpolated string TERMINATES it (CS1010/CS1003) ‚Äî never put a double-quote in an inline SQL comment inside `$@"‚Ä¶"` (escape as `""` or reword); and nested `$$"""‚Ä¶"""` raw-string interpolation to build test JSON is fragile (CS9007) ‚Äî build the JSON with plain string concatenation instead. *(from #226 ‚Äî a `"no data"` in the SQL comment broke the build; and #218 ‚Äî nested raw-string interpolation for the trace_signals seed JSON failed to compile.)*

- **A session-gated forensic read must be `Cache-Control: no-store` (or `private`), NEVER `public, max-age=N`.** `public` opts a shared cache into storing the Authorization-bearing 200; with `Vary: Origin` (not `Authorization`) it can serve that authed body to a later ANONYMOUS caller within the window, bypassing the gate (CWE-525). Match the reconcile / trace-signals siblings (`no-store`). *(from #218 review ‚Äî the error-diff endpoint shipped `public, max-age=15`; the Claude-review BLOCKED on it ‚Üí flipped to `no-store`.)*

- **Sandbox runs: INCLUDE them for FORENSIC/error signals (error-diff, flap-rate), EXCLUDE them for HEALTH rollups (SLO, availability, status, daily rollup).** A paused monitor's on-demand sandbox validations are the ONLY captures some checks have (canary 395 is entirely sandbox), so excluding them empties the forensic view; but they must never move availability. The test: is this a HEALTH signal (exclude sandbox, per 0064/0065) or a FORENSIC one (include)? *(from #218 error-diff and #223 flap-rate ‚Äî both deliberately count sandbox runs, the inverse of the SLO/status exclusion.)*

- **Current-state / status rollups must reflect the LAST SETTLED run, not the latest run.** Exclude `status='running'` from the latest-run pick and peel back to the most-recent settled run (mirror the dashboard's `lastSettledStatus`, dashboard #255): a failing check STAYS "down" while re-running (never drops to the "unknown" bucket, since `'running'` matches none of IsDownCritical/IsDegraded/up), a healthy one stays "up" (no blink), and only an all-`running` never-settled check is the ONE legitimate "unknown". Composes with the #278 superseded/awaiting exclusions (`superseded_by_run_id IS NULL AND status <> 'running' AND NOT(awaiting)`). *(from #226 ‚Äî `StatusFunctions` had no `running` exclusion, so a down property silently dropped out of "down" during every re-check while the dashboard component row on the SAME page still read "Down" ‚Äî an intra-page contradiction; live-caught on check 355 mid-run.)*

- **A key-less user-delegation blob SAS (minted via the API's MI) needs the `Storage Blob Delegator` role on the storage account ‚Äî `Storage Blob Data Reader` alone canNOT mint one** (`GetUserDelegationKey` 403s) ‚Äî plus a blob-service CORS rule for the browser's cross-origin fetch. Grant both in `infra/main.bicep`; the mint returns a graceful 503 until the role is live (deploy-safe). *(from #217 ‚Äî the SAS trace-serving cutover; the MI had only Data Reader, so the mint 503'd until the Delegator grant deployed.)*
