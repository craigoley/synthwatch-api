### OBSERVED — read auth model

- **Verb-gate default:** `AuthGate.Decide` returns `Allow` for every non-mutating verb — `AuthGate.cs:79` *"reads are always open (read-only default)"*. So GETs are open unless a handler self-guards.
- **Sensitive reads self-guard** via `SessionReadGate.RequireSessionAsync` (`SessionReadGate.cs:24-32`): resolves the caller with `IAuthPrincipal.FromBearerAsync(req.Headers.Authorization, ct)` → **401** if no valid session, **403** if live role isn't editor/admin (`CanWrite` floor).
- **The token is an opaque BEARER token**, not a cookie: `FromBearerAsync` → `AuthTokens.BearerFrom(authorizationHeader)` → `sessions` lookup by token hash → `{email, live role}` (`AuthPrincipalService.cs:34-48`). Role is re-resolved live on every request.

### INFERRED — MCP composability

Because auth is **`Authorization: Bearer <opaque-token>` with server-side session lookup and no cookie/CSRF/Origin dependency**, a headless MCP caller composes cleanly: mint a session token (OTP flow, or the admin-session recipe in `CLAUDE.md`) and forward it as a bearer header. The read surface is **two-tier**: the bulk (aggregates, status, metrics) is anon — wrappable with no token at all; the sensitive tier (channel creds, reconcile SQL, forensic artifacts) needs an editor/admin bearer. An MCP token model maps 1:1 onto the existing bearer/role floor — no cookie-session fight.

**Falsifier (run):** *"a read requires a browser cookie / CSRF token, so a headless bearer can't authenticate."* → `grep -rniE "Cookie|CSRF|SameSite|XSRF|antiforgery" Infrastructure/*.cs Functions/*.cs` → **no matches** (only bearer). Claim survives: reads are bearer-or-anon, headless-friendly. The one live check to fully confirm (not run here — no prod): `curl -H "Authorization: Bearer <token>" /api/channels` → 200; anon → 401 (the recipe already documented in `docs/auth-gates.md:52-58`).

### VERDICT — Q1

**The read surface is complete + stable enough to wrap as MCP tools for most queries WITHOUT new endpoints — with two gaps, one of them load-bearing.**

Already answerable via existing endpoints:
- **"p95 for check X over 7d"** → `GET /checks/{id}/metrics` (carries p50/p95) or `GET /reports/performance?window=7d` (per-check p95, `groupBy`). ✅ the *check-X / 7d* part.
- Availability/uptime %, SLA headline, MTTR, incident breakdown, trust/regression scorecard, deploy list, region freshness, per-run history — all direct GETs.

Would need a NEW endpoint (or client-side MCP orchestration):
- **"…in env Y"** — ❌ **no environment dimension exists anywhere** (see Q2 falsifier). No GET accepts `?env`; no `environment` column on checks/runs. This is not wrappable — it needs the schema/dimension work in Q2 first.
- **"which monitors regressed near the last deploy"** — ⚠️ *composable but not single-shot*: `GET /reports/deploys` gives deploy times and `GET /incidents/{id}` carries `nearbyDeploys` (#157), and per-check regression is in `/reports/trust`, but **no single endpoint returns "monitors whose p95/status regressed within N min of the last deploy."** An MCP tool can orchestrate it from 2-3 calls; a first-class endpoint would be a convenience, not a prerequisite.

**Bottom line:** wrap-ready for the fleet/check/window/incident/report queries today (bearer or anon). The **"env Y" axis is the one true blocker** and is a data-model gap (Q2), not an endpoint-shape gap.
