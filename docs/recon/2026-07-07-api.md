### OBSERVED ‚Äî read auth model

- **Verb-gate default:** `AuthGate.Decide` returns `Allow` for every non-mutating verb ‚Äî `AuthGate.cs:79` *"reads are always open (read-only default)"*. So GETs are open unless a handler self-guards.
- **Sensitive reads self-guard** via `SessionReadGate.RequireSessionAsync` (`SessionReadGate.cs:24-32`): resolves the caller with `IAuthPrincipal.FromBearerAsync(req.Headers.Authorization, ct)` ‚Üí **401** if no valid session, **403** if live role isn't editor/admin (`CanWrite` floor).
- **The token is an opaque BEARER token**, not a cookie: `FromBearerAsync` ‚Üí `AuthTokens.BearerFrom(authorizationHeader)` ‚Üí `sessions` lookup by token hash ‚Üí `{email, live role}` (`AuthPrincipalService.cs:34-48`). Role is re-resolved live on every request.

### INFERRED ‚Äî MCP composability

Because auth is **`Authorization: Bearer <opaque-token>` with server-side session lookup and no cookie/CSRF/Origin dependency**, a headless MCP caller composes cleanly: mint a session token (OTP flow, or the admin-session recipe in `CLAUDE.md`) and forward it as a bearer header. The read surface is **two-tier**: the bulk (aggregates, status, metrics) is anon ‚Äî wrappable with no token at all; the sensitive tier (channel creds, reconcile SQL, forensic artifacts) needs an editor/admin bearer. An MCP token model maps 1:1 onto the existing bearer/role floor ‚Äî no cookie-session fight.

**Falsifier (run):** *"a read requires a browser cookie / CSRF token, so a headless bearer can't authenticate."* ‚Üí `grep -rniE "Cookie|CSRF|SameSite|XSRF|antiforgery" Infrastructure/*.cs Functions/*.cs` ‚Üí **no matches** (only bearer). Claim survives: reads are bearer-or-anon, headless-friendly. The one live check to fully confirm (not run here ‚Äî no prod): `curl -H "Authorization: Bearer <token>" /api/channels` ‚Üí 200; anon ‚Üí 401 (the recipe already documented in `docs/auth-gates.md:52-58`).

### VERDICT ‚Äî Q1

**The read surface is complete + stable enough to wrap as MCP tools for most queries WITHOUT new endpoints ‚Äî with two gaps, one of them load-bearing.**

Already answerable via existing endpoints:
- **"p95 for check X over 7d"** ‚Üí `GET /checks/{id}/metrics` (carries p50/p95) or `GET /reports/performance?window=7d` (per-check p95, `groupBy`). ‚úÖ the *check-X / 7d* part.
- Availability/uptime %, SLA headline, MTTR, incident breakdown, trust/regression scorecard, deploy list, region freshness, per-run history ‚Äî all direct GETs.

Would need a NEW endpoint (or client-side MCP orchestration):
- **"‚Ä¶in env Y"** ‚Äî ‚ùå **no environment dimension exists anywhere** (see Q2 falsifier). No GET accepts `?env`; no `environment` column on checks/runs. This is not wrappable ‚Äî it needs the schema/dimension work in Q2 first.
- **"which monitors regressed near the last deploy"** ‚Äî ‚öÝÔ∏è *composable but not single-shot*: `GET /reports/deploys` gives deploy times and `GET /incidents/{id}` carries `nearbyDeploys` (#157), and per-check regression is in `/reports/trust`, but **no single endpoint returns "monitors whose p95/status regressed within N min of the last deploy."** An MCP tool can orchestrate it from 2-3 calls; a first-class endpoint would be a convenience, not a prerequisite.

**Bottom line:** wrap-ready for the fleet/check/window/incident/report queries today (bearer or anon). The **"env Y" axis is the one true blocker** and is a data-model gap (Q2), not an endpoint-shape gap.

## Q2 ‚Äî ‚òÖS1 SLO/report environment-partitionability

### OBSERVED ‚Äî the SLO aggregation grouping keys

`GET /reports/slo` (`ReportsFunctions.cs:184-198`):
```sql
FROM checks c
CROSS JOIN LATERAL slo_status(c.id, {from}, {to}) s      -- per-check budget/burn, window args
CROSS JOIN LATERAL slo_burn_status(c.id) b               -- per-check page-state (fixed 1h/6h/30m)
WHERE c.slo_target IS NOT NULL
  AND (cardinality({tags}) = 0 OR c.id IN (... check_tags AND-match ...))
ORDER BY c.name
```
- **The only grouping key is `check_id`** (one row per SLO-having check). `slo_status(c.id, from, to)` and `slo_burn_status(c.id)` are runner-owned DB functions keyed **solely on `check_id` + window** ‚Äî no env argument at any layer.
- The fleet rollup is `SloReportProjection.Build(rows)` over **every** returned row (all `slo_target IS NOT NULL` checks).
- The **only partition dimension that exists is `tag`** (`?tag=key:value`, repeatable), and it is **include-only** (`c.id IN (checks having ALL selected tags)`) ‚Äî there is no exclude/NOT-tag path.
- Same shape in `/reports/mttr` (`ReportsFunctions.cs:238-248`), `/reports/incident-breakdown`, `/reports/trust`: `FROM checks/incidents ‚Ä¶ JOIN checks ‚Ä¶ [optional AND-tag include]`, grouped by `check_id`. No env dimension.

### OBSERVED ‚Äî the environment falsifier

**Falsifier (run):** *"there is an `environment` dimension the aggregation could exclude on."*
`grep -rniE "environment|\benv\b" Data/Entities/*.cs Data/SynthWatchDbContext.cs` ‚Üí the **only** hit is `EnvironmentRegional` (`ReportingRows.cs:97`, `SynthWatchDbContext.cs:468`), which is an **RCA classification bucket**, not a deploy env: `count(*) FILTER (WHERE i.rca->>'classification' = 'environment-regional')` (`ReportsFunctions.cs:407`) ‚Äî one of {real-outage, flaky-transient, selector-drift, environment-regional, perf-regression, unclassified}. Fixture schema has **no** `environment` column on `checks`/`runs`. ‚Üí Falsifier fails to find an env dimension: **none exists.**

### VERDICT ‚Äî Q2

**Env-partition is NOT a pure filter addition, but also NOT a full rearchitecture ‚Äî it sits in between, and the cleanest form is a small schema add + a default-exclude, because the aggregation is keyed on `check_id` only with `tag` as the sole (include-only) partition axis.**

Three honest options, cheapest‚Üísoundest:
1. **Tag-as-env (no schema change), but blocked by include-only semantics.** Tags already partition (`env:preprod`). But today's filter can only *include* checks with a tag ‚Äî to keep pre-prod OUT of the **default (no-tag) prod fleet SLO**, you'd need a **NOT-tag / default-exclude** clause. That's a **query/filter addition** (add a negative-tag predicate + make prod aggregations default to `env != preprod`), a few lines per report query ‚Äî no schema, no runner change. Weakness: env-as-tag is a soft convention (a mistagged/untagged pre-prod check silently pollutes prod).
2. **First-class `checks.environment` column (runner-owned schema).** A real dimension the default prod aggregation excludes on (`WHERE coalesce(environment,'prod')='prod'`). This is a **runner migration** (the runner owns the schema ‚Äî see `CLAUDE.md`) + read-side filter here. Soundest; not a rearchitecture (the aggregation already groups per-check ‚Äî you're adding a WHERE, not reshaping the rollup).
3. Neither `slo_status`/`slo_burn_status` needs restructuring ‚Äî both already take `check_id`; env-exclusion happens in the *outer* `FROM checks` predicate, before the LATERAL. So the burn/page path the runner shares is untouched.

**Load-bearing caveat:** because the fleet rollup today includes **every** `slo_target IS NOT NULL` check with no env filter, **a pre-prod check with an SLO target pollutes the prod budget the moment it exists** ‚Äî the partition must be *default-exclude*, not opt-in, to actually protect prod. Option 1's include-only tag filter does **not** achieve that without the added negative predicate.
