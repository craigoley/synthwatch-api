## Q1 — how a plan gets APPROVED (does the executor need more than the flag?)

`POST /api/reconcile/approve {sourceKey, driftType}` → `DecideAsync` (`ReconcileFunctions.cs:152-184`).
It is **not** a bare flag flip — it enforces three guards then writes the flag + audit:
- **blocked** (a redaction strip) → 409 (`:165`). *(140 is `new`, not blocked.)*
- **executability** — only `ApplyExecutableDriftTypes` may be approved; else 409 (`:171`).
  `ApplyExecutableDriftTypes = {new, missing, changed}` (`:131`), so `new` passes.
- **pending** — status must be `pending`, else 409 (`:174`).
- Then: `UPDATE … SET status='approved', decided_at=now(), decided_by={who}` **race-gated on
  `status='pending'`** (`:179-181`); `who` = the authenticated principal's email (`:177`).

**Is a raw SQL flip safe?** The executor (apply) selects `WHERE status='approved'` only — it does **not**
read `decided_by`/`decided_at` (`:200`). So a raw `UPDATE … SET status='approved'` **would** be picked
up. But the API is preferred: it enforces the guards (moot for 140, but correct), stamps the audit
fields, and is race-safe. **Approve does nothing else** — no side effects, no auto-apply.

## Q2 — how an approved plan gets APPLIED (auto or separate step?)

**A SEPARATE, MANUAL step:** `POST /api/reconcile/apply` (`ReconcileFunctions.cs:186-…`). Not
automatic-on-approve; **no cron** invokes it (the runner's `reconcileMain` is detect-only; no
`.github/workflows` calls apply). It:
- selects `WHERE status='approved' AND drift_type = ANY({new,missing,changed})`, **cap 5** (`ApplyCap`) (`:198-201`),
- per plan, in ONE transaction: `new` → INSERT the check (sensitive inline, **enabled=FALSE**) + seed
  `check_locations` for every enabled region, then flip the plan `approved→applied` (`:281-282`),
- per-plan try/catch (one bad plan rolls back alone; others proceed).

So the path is **two manual API calls: approve, then apply.**
